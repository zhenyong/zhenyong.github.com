如何跟踪仓库子目录之 subtree 命令

2016-07-20
tech
git

想翻译 react 的网站，然后发现 github 仓库中，有关网站部分的源码放在子目录 /docs 下，于是想能否单独 folk 子目录 /docs，而且以后只 pull 这个子目录的更新，于是提交记录就只有该目录下的，我就能专心对比提交然后修改对应翻译。

通过阅读下列资源

  • Forking a sub directory … part of my own repo
  • Using Git subtrees to split a repository
  • git-subtree/git-subtree.txt

找到解决方法：

最终的效果就是：我的 zhenyong/react-website-cn 仓库的 /docs 目录跟踪 react 仓库的子目录 /docs，具体工作流有两种，下面分别讲解

1. 工作在 react 仓库

下面介绍的操作都工作在 react 的仓库目录

first push

归纳：

  • clone react
  • -> split 子目录成为新分支
  • -> push 我的库

follow update

方法21

接下来，如果 react 团队更新了提交，我要怎么更新 docs-branch 分支呢,
简单的思路就是，回到 master 分支，重复上面的操作：

归纳：

  • master pull
  • -> split 子目录成为分支
  • -> push 我的库

方法 2

当然，你估计会想，为什么不直接在 docs-branch 分支中更新了，当然是可以的，不过要注意，这里涉及两个远程库，一个是 react 库 (origin)，一个是我们的库 (zhenyong)，拉取和推送使用不同的库：

归纳：

  • subtree pull react库
  • -> push 我的库

总结

上面的操作演示了如果持续的把 react 仓库子目录 /docs 最新提交同步推送到我们的库。

实际使用时，你肯定还要修改自己库的代码，下面介绍在自己仓库目录下的工作。

2. 工作在自己的仓库

假设我的库是一个全新的，空的仓库，我希望我有一个分支 docs-branch
专门用来跟踪 react 远程库的 docs 子目录，然后 master 分支则
时刻包含 (merge/rebase) docs-branch 分支，并且在 master 增加中文文档目录 docs-cn，这样就不会有冲突。

归纳:

  • clone 我的库
  • -> 分支 docs-branch
  • -> 跟踪 react 库子目录
  • -> master 融合 docs-branch

注意： 如果你嫌 react /docs 目录下提交记录太多，污染你的仓库，你可以考虑在
subtree 的 add splite pull 命令时带上 --squash 参数，这样
会将 react 仓库的多条记录当做一条记录在你的仓库中，个人觉得首次这么弄可以，
后续更新还是保持多条，也不容易引起冲突。

总结

通过在两个仓库目录的工作方式，介绍 git subtree 命令的特点，让你更好的 folk 子目录。

实际工作起来，我会在自己仓库目录下玩，当然你在本地 react 仓库目录里创建一个分支跟踪 zhenyong 库也是一回事，融会贯通嘛。
